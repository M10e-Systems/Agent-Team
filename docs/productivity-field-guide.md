# Productivity Field Guide

## Premise

Agent teams can amplify personal productivity by turning recurring modes of thinking into stable, addressable collaborators.

That lets you keep final authority while still getting the benefit of different perspectives, different paces, and different forms of accountability.

## Field Use Cases

- Solo founder or product operator: facilitator, strategist, growth, operations
- Creative production: editor, maker, critic, distributor
- Research and synthesis: scout, skeptic, synthesizer, action planner
- Personal execution: planner, blocker remover, accountability partner, writing assistant
- Community or Discord room: visible specialist bots that help steer a project channel

## Personal Productivity Patterns

- Ask the facilitator to shape the problem before asking specialists to answer it
- Use one specialist voice for a focused question instead of a noisy group prompt
- Use a room round when you want contrasting lenses on the same problem
- Keep the team small at first so the added structure improves clarity instead of creating ceremony
- Reuse the same roles over time so the team becomes easier to address and easier to trust

## Operating Principles

- The human remains the decision owner
- Agents should not take outward action without authorization
- Shared memory should be intentional and minimal
- Role boundaries should reduce noise, not create bureaucracy
- Track what changed in the real world, not just what the agents said

## Example Team Shapes

- Launch team: facilitator, architect, growth
- Content team: editor, maker, distributor
- Research team: scout, skeptic, synthesizer
- Personal execution team: planner, blocker remover, accountability partner

These are examples, not requirements. The value comes from choosing a small set of stable roles that fit the actual work.

## Risks And Guardrails

- Too many roles can create overhead instead of leverage
- Shared memory can become noisy if everything is treated as durable
- A team can sound productive without actually moving the work forward
- Response obligation should be reserved for cases where the role truly needs to answer

Guardrails:

- start with a narrow scope
- keep the contracts short
- review what was actually shipped or decided
- keep one human in charge of final decisions

## What To Try First

1. Pick one recurring project.
2. Define three roles that reflect the real work.
3. Write a short shared contract for how those roles should speak.
4. Route one room prompt and one direct ask through the team.
5. Keep what improved clarity and remove what added noise.
